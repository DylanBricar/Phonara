## Why this matters

`bun audit` reports **22 advisories (13 high, 9 moderate)**. All of them are in
**dev / build-time** transitive dependencies — `vite`, `rollup`, `esbuild`,
`picomatch`, `flatted`, `minimatch`, `lodash` (via `eslint-plugin-i18next`) —
none ship in the desktop binary. So the runtime risk to end users is low, but the
advisories include real developer-machine hazards (Vite dev-server arbitrary file
read, Rollup path-traversal file write, esbuild missing-integrity RCE) and the
noise makes `bun audit` useless as a signal until it's cleaned up. Most are
fixable by updating within the existing semver ranges.

## Scope

**In scope** (modify only these):
- `bun.lock` (regenerated by `bun update`)
- `package.json` — only if a direct dependency's range must be raised to pull a
  fixed transitive version (see Step 3)

**Out of scope** (do NOT touch):
- Application source. This plan changes dependencies only.
- `src-tauri/Cargo.toml` / Rust deps — a separate concern (not covered here).
- Do NOT run `bun update --latest` blanket-style — that pulls major versions and
  can break the build. Use it surgically only as described in Step 3.

## Steps

### Step 1: Baseline

Run `bun audit` and record the advisory count (expected: 22 — 13 high, 9
moderate). This is the before-number.

### Step 2: Update within existing ranges

Run `bun update` (updates everything to the newest versions **allowed by the
current `package.json` ranges**). This alone should clear the advisories whose
fix lives within range (e.g. `vite`/`rollup`/`esbuild`/`picomatch` patch bumps).

**Verify**:
- `bun run build` → exit 0 (tsc passes, vite build produces `dist/`).
- `bun run lint` → exit 0.
- `bun audit` → strictly fewer advisories than baseline.

If `bun run build` fails, the most likely cause is a Vite 6 minor bump changing
behavior — STOP and report rather than pinning blindly.

### Step 3: Address advisories still flagged (range-bump only the direct deps)

Re-run `bun audit`. For each remaining advisory, find the **direct** dependency
that pulls it (the audit output shows the dependency path, e.g.
`vite › rollup`, `eslint-plugin-i18next › lodash`). For each:

- If a newer **non-breaking** (same major) version of the direct dep exists that
  resolves it, raise that dep's range in `package.json` (e.g. `vite: ^6.4.1` →
  `^6.5.0`) and re-run `bun update`.
- If the only fix requires a **major** bump of a direct dep (e.g.
  `eslint-plugin-i18next` major, or `@typescript-eslint` major), do **NOT** do it
  under this plan — record it in your report as "requires major upgrade, deferred"
  with the advisory and the version that would fix it.

After each change: `bun run build` → 0, `bun run lint` → 0.

**Verify**: `bun audit` → all *fixable-within-major* advisories cleared; any
remainder is documented as major-upgrade-required.

### Step 4: Summarize

In your final report, list: advisories cleared, advisories remaining (with the
reason: "needs major bump of X"), and confirm `bun run build` + `bun run lint`
still pass.
